Guia de Deploy

🚀 LIDDIS — Do Zero ao Ar

Passo a passo completo para quem nunca fez deploy: criar contas, configurar serviços e publicar o sistema.

⏱️Tempo estimado: 2–3 horas
💡Nível: Iniciante
💰Custo inicial: Gratuito (dentro dos limites free)
🖥️OS: Windows / Mac / Linux
📌

Antes de começar, separe:

  • Uma conta Google (Gmail) que será usada para o Google Cloud
  • Um bloco de notas ou gerenciador de senhas para salvar chaves e URLs
  • Conexão estável com a internet

Instalar Ferramentas no seu Computador

Pré-requisito

Instalar o Python (se ainda não tiver)

Acesse python.org/downloads, baixe a versão mais recente e instale.

⚠️

Durante a instalação, marque a opção "Add Python to PATH" antes de clicar em Install.

Verifique se funcionou abrindo o terminal e digitando:

python --version

Instalar o Google Cloud CLI (gcloud)

É a ferramenta que usaremos para enviar o sistema para o Google Cloud.

  1. Acesse: cloud.google.com/sdk/docs/install
  2. Escolha seu sistema operacional e baixe o instalador
  3. Execute o instalador e siga as telas (Next → Next → Install)
  4. Ao final, o instalador vai abrir um terminal automaticamente — feche-o

Verifique se funcionou:

gcloud --version

Instalar o Git (se ainda não tiver)

Acesse git-scm.com, baixe e instale. Todas as opções padrão estão corretas.

1

Supabase — Banco de Dados

~15 minutos
ℹ️

O Supabase vai guardar todos os dados do sistema (pacientes, consultas, usuários). É gratuito para projetos pequenos e fica em servidores no Brasil.

Criar conta no Supabase

  1. Acesse supabase.com
  2. Clique em Start your project
  3. Clique em Continue with GitHub (ou crie uma conta com e-mail)
  4. Autorize o acesso quando solicitado

Criar o projeto do banco de dados

  1. No painel do Supabase, clique em New project
  2. Preencha assim:
Name liddis-producao
Database Password Gere uma senha forte — salve em local seguro! Não tem como recuperar depois.
Region South America (São Paulo) — escolha essa para menor latência no Brasil
  1. Clique em Create new project
  2. Aguarde cerca de 2 minutos enquanto o projeto é criado

Copiar a URL de conexão do banco

Essa URL é o endereço que o sistema usa para se conectar ao banco.

  1. Dentro do projeto, clique no ícone de engrenagem ⚙️ (Project Settings) no menu lateral
  2. Clique em Database
  3. Role a página até encontrar Connection string
  4. Clique na aba URI
  5. Localize a seção Connection Pooler (importante!)
  6. Copie a URL — ela terá este formato:
postgresql://postgres.[REF]:[SUA_SENHA]@aws-0-sa-east-1.pooler.supabase.com:6543/postgres
⚠️

Use a porta 6543 (Connection Pooler), não a porta 5432. Isso é essencial para funcionar no Cloud Run.

📋 Salve essa URL — você vai precisar dela mais adiante.

2

Google Cloud — Criar Conta e Projeto

~20 minutos
ℹ️

O Google Cloud hospeda o sistema online. O plano gratuito inclui créditos iniciais de $300 por 90 dias e um nível gratuito permanente para o Cloud Run.

Criar conta no Google Cloud

  1. Acesse console.cloud.google.com
  2. Faça login com sua conta Google (Gmail)
  3. Aceite os termos de serviço
  4. Quando perguntado sobre Free Trial, clique em Activate — será necessário um cartão de crédito para verificação, mas não será cobrado nada automaticamente
✅

O cartão é usado apenas para verificar identidade. Você recebe $300 de créditos e o Cloud Run tem um nível gratuito permanente.

Criar um novo projeto

  1. No topo da tela, clique no seletor de projeto (ao lado do logo do Google Cloud)
  2. Clique em New Project
  3. Preencha:
    • Project name: liddis
    • Organization: deixe como está (No organization)
  4. Clique em Create
  5. Aguarde alguns segundos e selecione o projeto criado

📋 Anote o Project ID — aparece abaixo do nome do projeto (ex: liddis-123456). Você vai usar em vários comandos.

Fazer login com o gcloud no terminal

Abra o terminal (CMD ou PowerShell no Windows, Terminal no Mac) e execute:

gcloud auth login

Uma janela do navegador vai abrir — faça login com a mesma conta Google. Depois execute:

gcloud config set project SEU-PROJECT-ID
⚠️

Substitua SEU-PROJECT-ID pelo ID anotado no passo anterior (ex: liddis-123456).

Habilitar os serviços necessários

Este comando ativa os serviços do Google Cloud que o LIDDIS precisa. Execute no terminal:

gcloud services enable run.googleapis.com cloudbuild.googleapis.com artifactregistry.googleapis.com secretmanager.googleapis.com

Aguarde até aparecer a mensagem de sucesso. Pode levar 1–2 minutos.

Criar repositório para as imagens Docker

O sistema é empacotado em uma "imagem Docker" antes de ser publicado. Execute:

gcloud artifacts repositories create liddis --repository-format=docker --location=southamerica-east1 --description="Imagens Docker do LIDDIS"
3

Google OAuth — Login com Google

~15 minutos
ℹ️

O OAuth permite que os usuários do LIDDIS façam login usando a conta Google deles, sem precisar criar uma senha separada.

Configurar a tela de consentimento OAuth

  1. No Google Cloud Console, clique no menu ☰ (três linhas no topo esquerdo)
  2. Vá em APIs & Services → OAuth consent screen
  3. Selecione External e clique em Create
  4. Preencha:
    • App name: LIDDIS
    • User support email: seu e-mail
    • Developer contact email: seu e-mail
  5. Clique em Save and Continue em todas as próximas telas (sem mudar nada)
  6. Na última tela, clique em Back to Dashboard

Criar as credenciais OAuth

  1. Ainda em APIs & Services, clique em Credentials
  2. Clique em + Create Credentials → OAuth 2.0 Client IDs
  3. Em Application type, selecione Web application
  4. Em Name, coloque: LIDDIS Web

Configurar as URLs autorizadas

Ainda na mesma tela de criação das credenciais:

Authorized JavaScript origins — clique em "+ Add URI" e adicione:

http://localhost:8000

Authorized redirect URIs — clique em "+ Add URI" e adicione:

http://localhost:8000/accounts/google/login/callback/
📌

Depois do deploy você vai adicionar a URL real do Cloud Run. Por enquanto, coloque apenas o localhost para testar localmente.

  1. Clique em Create
  2. Uma janela vai aparecer com o Client ID e o Client Secret
  3. 📋 Copie e salve ambos agora — o secret não aparece novamente!
4

Resend — Envio de E-mails

~10 minutos
ℹ️

O Resend envia os e-mails de verificação de conta para os usuários. O plano gratuito suporta 100 e-mails/dia.

Criar conta no Resend

  1. Acesse resend.com
  2. Clique em Sign Up e crie uma conta (pode usar GitHub ou e-mail)
  3. Verifique seu e-mail se solicitado

Criar uma API Key

  1. No painel do Resend, clique em API Keys no menu lateral
  2. Clique em Create API Key
  3. Em Name, coloque: liddis-prod
  4. Deixe as permissões padrão e clique em Add
  5. 📋 Copie a API Key gerada — começa com re_
⚠️

A API Key só aparece uma vez! Salve imediatamente no seu bloco de notas.

5

Secret Manager — Guardar Senhas com Segurança

~15 minutos
ℹ️

O Secret Manager guarda as senhas e chaves do sistema de forma segura no Google Cloud. Nunca deixe senhas no código!

Gerar uma SECRET_KEY forte para o Django

Execute este comando no terminal para gerar uma chave aleatória segura:

python -c "import secrets; print(secrets.token_urlsafe(50))"

📋 Copie e salve a chave gerada.

Criar os secrets no Google Cloud

Execute cada comando abaixo no terminal, substituindo os valores pelos que você salvou:

Secret Key do Django:

echo -n "SUA_SECRET_KEY_AQUI" | gcloud secrets create liddis-secret-key --data-file=-

URL do banco Supabase:

echo -n "postgresql://postgres.[REF]:[SENHA]@aws-0-sa-east-1.pooler.supabase.com:6543/postgres" | gcloud secrets create liddis-database-url --data-file=-

Client ID do Google OAuth:

echo -n "SEU_CLIENT_ID.apps.googleusercontent.com" | gcloud secrets create liddis-google-client-id --data-file=-

Client Secret do Google OAuth:

echo -n "GOCSPX-SEU_SECRET_AQUI" | gcloud secrets create liddis-google-secret --data-file=-

API Key do Resend:

echo -n "re_SUA_API_KEY_AQUI" | gcloud secrets create liddis-resend-key --data-file=-

Dar permissões ao Cloud Build para acessar os secrets

Execute esses comandos em sequência no terminal:

PROJECT_NUMBER=$(gcloud projects describe $(gcloud config get-value project) --format='value(projectNumber)') SA="$PROJECT_NUMBER@cloudbuild.gserviceaccount.com" gcloud projects add-iam-policy-binding $(gcloud config get-value project) --member="serviceAccount:$SA" --role="roles/run.admin" gcloud projects add-iam-policy-binding $(gcloud config get-value project) --member="serviceAccount:$SA" --role="roles/secretmanager.secretAccessor" gcloud projects add-iam-policy-binding $(gcloud config get-value project) --member="serviceAccount:$SA" --role="roles/iam.serviceAccountUser"

Preparar o Banco de Dados

~10 minutos

Abrir a pasta do projeto no terminal

No Windows, abra o terminal na pasta do projeto. Se você usar VS Code, pressione Ctrl + ` para abrir o terminal integrado.

Configurar o arquivo .env de produção

  1. Abra o arquivo dev-env/.env.production no VS Code
  2. Preencha cada variável com os valores que você salvou:
Variável Valor Req.
SECRET_KEY A chave gerada no passo anterior Sim
DATABASE_URL URL do Supabase (porta 6543) Sim
GOOGLE_CLIENT_ID Client ID do Google OAuth Sim
GOOGLE_SECRET Client Secret do Google OAuth Sim
RESEND_API_KEY API Key do Resend (re_...) Sim
DEBUG False Sim

Copiar o .env de produção e aplicar as migrações

No terminal (na pasta do projeto), execute:

# Windows — copiar o .env copy dev-env\.env.production .env # Mac/Linux cp dev-env/.env.production .env

Agora aplique as migrações no banco Supabase:

python manage.py migrate

Crie o administrador do sistema:

python manage.py createsuperuser

O terminal vai pedir nome de usuário, e-mail e senha — preencha e guarde essas informações.

Publicar o Sistema (Deploy)

~20 minutos

Fazer o primeiro deploy manual

Na pasta raiz do projeto, execute:

gcloud builds submit --config cloudbuild.yaml

Esse processo leva entre 5 e 10 minutos. Você vai ver logs rolando no terminal — é normal.

✅

Quando aparecer "Finished successfully", o deploy foi concluído!

Descobrir a URL do seu sistema

gcloud run services describe liddis-backend --region=southamerica-east1 --format="value(status.url)"

A URL terá um formato como: https://liddis-backend-xxxxxxxx-rj.a.run.app

📋 Salve essa URL! Você vai precisar dela para configurar o OAuth.

Adicionar a URL real no Google OAuth

  1. Volte ao Google Cloud Console → APIs & Services → Credentials
  2. Clique no lápis ✏️ para editar o OAuth Client ID criado anteriormente
  3. Em Authorized JavaScript origins, adicione sua URL real:
https://liddis-backend-xxxxxxxx-rj.a.run.app
  1. Em Authorized redirect URIs, adicione:
https://liddis-backend-xxxxxxxx-rj.a.run.app/accounts/google/login/callback/
  1. Clique em Save

Configurar o Site no painel de administração

  1. Acesse https://SUA-URL.run.app/admin/
  2. Faça login com o superusuário criado no passo 6
  3. Clique em Sites
  4. Clique no site existente (example.com)
  5. Mude para:
Domain name Sua URL sem o https:// (ex: liddis-backend-xxx.a.run.app)
Display name LIDDIS
  1. Clique em Save

Configurar o Google OAuth no painel

  1. Ainda no Admin, procure por Social Applications
  2. Clique em Add Social Application
  3. Preencha:
Provider Google
Name Google OAuth
Client ID Cole o Client ID do Google
Secret Key Cole o Client Secret do Google
Sites Mova o site da caixa da esquerda para a da direita clicando nele e depois em →
  1. Clique em Save

Deploy Automático a cada Push (opcional)

~10 minutos
ℹ️

Após configurar isso, toda vez que você enviar código para o GitHub na branch main, o sistema será atualizado automaticamente, sem precisar rodar comandos.

Conectar o repositório GitHub ao Cloud Build

  1. No Cloud Console, vá no menu ☰ → Cloud Build → Triggers
  2. Clique em Create Trigger
  3. Clique em Connect Repository
  4. Selecione GitHub (Cloud Build GitHub App)
  5. Autorize o acesso ao GitHub quando solicitado
  6. Selecione o repositório do LIDDIS e clique em Connect

Configurar o trigger de deploy

  1. De volta na tela de criação do trigger, preencha:
Name deploy-main
Event Push to a branch
Branch ^main$
Configuration Cloud Build configuration file → cloudbuild.yaml
  1. Clique em Save
✅

Pronto! Agora cada git push origin main faz o deploy automático.

Problemas Comuns e Soluções

Troubleshooting

Erro: "gcloud: command not found"

O Google Cloud CLI não foi instalado corretamente ou não está no PATH. Reinstale seguindo o guia oficial e reinicie o terminal.

Erro de conexão com o banco (OperationalError)

Verifique se a DATABASE_URL usa a porta 6543 (Connection Pooler) e não 5432. Confirme que a senha está correta.

Login com Google redireciona para erro 400

A URL de callback não está cadastrada no Google OAuth. Certifique-se que Authorized redirect URIs inclui exatamente: https://SUA-URL.run.app/accounts/google/login/callback/ (com a barra no final).

O sistema abre mas dá erro 500

Veja os logs para entender o erro:

gcloud run logs tail liddis-backend --region=southamerica-east1

Migrations falharam

Verifique se o arquivo .env está na raiz do projeto (não dentro de dev-env) e se DATABASE_URL está preenchida corretamente.
