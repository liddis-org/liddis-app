LIDDIS

Transformando Dados em Inteligência Clínica

Plataforma web para gestão de saúde pessoal e integração entre pacientes e profissionais de saúde. Permite o registro de consultas, sinais vitais e prontuários vinculados via token seguro de atendimento.

🐍 Python 3.10
⚙ Django 5.2
🔐 JWT + Session Auth
🗄 SQLite → PostgreSQL
📡 REST Framework
⚙

Fluxo de Atendimento por Token

Como funciona a sessão segura entre paciente e profissional

Lado do Paciente

1

Acessa "Iniciar Atendimento"

No dashboard ou sidebar, o paciente clica em Iniciar Atendimento → /consultas/atendimento/iniciar/

2

Token UUID é gerado

O sistema cria uma ConsultationSession com status pending, token UUID único e validade de 2 horas. Sessões antigas são canceladas.

3

Mostra o código na tela

O paciente vê o código completo (UUID) e o exibe ao profissional de saúde fisicamente.

Lado do Profissional (Médico, Enfermeiro, etc.)

1

Acessa "Atender Paciente"

No dashboard do profissional → /consultas/atendimento/entrar/

2

Insere o token

Cola o UUID recebido do paciente. O sistema valida: token existe, status é pending e não está expirado.

3

Acessa dados e registra a consulta

Vê o perfil básico do paciente (nome, idade, e-mail) e preenche diagnóstico, prescrição e anotações.

4

Sessão é encerrada automaticamente

Ao salvar, a consulta é vinculada ao paciente, a sessão muda para closed e o token não pode mais ser reutilizado.

config/settings.py

Configurações centrais do Django

⚙
settings.py
config/settings.py
Crítico

Arquivo central de configuração do Django. Define banco de dados, apps instalados, autenticação, middlewares e variáveis de segurança carregadas do arquivo .env via python-decouple.

SECRET_KEY — chave criptográfica lida do .env, nunca exposta no código
DEBUG — True em desenvolvimento, False em produção
AUTH_USER_MODEL — aponta para users.CustomUser (modelo de usuário customizado)
DATABASES — SQLite por padrão; variáveis prontas para PostgreSQL
AUTHENTICATION_BACKENDS — define ModelBackend explicitamente para evitar ambiguidade
INSTALLED_APPS — inclui users, consultations, rest_framework, corsheaders
Sessões — SESSION_COOKIE_HTTPONLY=True, CSRF_COOKIE_HTTPONLY=True por segurança

⚠ Para trocar para PostgreSQL: defina DB_ENGINE, DB_NAME, DB_USER, DB_PASSWORD, DB_HOST, DB_PORT no .env

ConsultationSession
consultations/models.py · class ConsultationSession
ModelSegurança

Controla o ciclo de vida de uma sessão de atendimento via token UUID. Um paciente gera o token, um profissional o consome. Após o uso, a sessão é encerrada e o token não pode ser reutilizado.

CampoTipoDescrição
tokenUUIDFieldIdentificador único, gerado automaticamente, não editável
patientFK→UserPaciente que gerou o token
professionalFK→User (null)Profissional que usou o token (preenchido ao entrar)
consultationOneToOne→Consultation (null)Consulta gerada ao final do atendimento
statusCharFieldpending / active / closed / expired
expires_atDateTimeFieldToken expira 2h após criação
is_expired — property: True se now() > expires_at
is_valid — property: True se pending e não expirado
token_display — property: primeiros 8 caracteres em maiúsculo (exibição)
